# 前端路由权限与按钮权限

> JeeSite.NET 的前端权限体系包含**路由级权限**（能否访问某页面）、**按钮级权限**（能否看到/点击某按钮）、**字段级权限**（某些字段是否显示/是否可编辑）。

---

## 一、前端权限体系概览

### 三层权限

```
┌───────────────────────────────────────────────────┐
│  第一层：路由级权限（Router Guard）                │
│  ├── 登录态校验（无 token → 重定向 /login）       │
│  ├── 路由 meta.permission 校验（无权限 → /403）  │
│  └── 动态菜单路由注册（从后端拉菜单 → 自动 addRoute）│
├───────────────────────────────────────────────────┤
│  第二层：按钮级权限（Directive + Composable）      │
│  ├── v-permission="'sys:user:add'"                │
│  ├── v-permission:any="['cms:article:edit', ...]" │
│  └── const perm = usePermission(); perm.has(...) │
├───────────────────────────────────────────────────┤
│  第三层：字段级权限（Dynamic Form Schema）         │
│  ├── hidden: 字段不渲染                           │
│  ├── disabled: 字段不可编辑                       │
│  └── 后端接口同步过滤（服务端双重保障）           │
└───────────────────────────────────────────────────┘
```

### 权限数据来源

前端在登录后调用 `GET /api/v1/sys/auth/menu-route`，返回以下三部分：

| 部分 | 格式 | 说明 |
|------|------|------|
| `permissions` | `string[]` | 权限标识数组（如 `sys:user:add`） |
| `menus` | 树状 `Menu[]` | 菜单树，含 name/path/component/permission/children |
| `fieldPermissions` | `Dictionary<string, FieldPermission[]>` | 每个页面的字段权限 |

### 权限存储

- 位置：Pinia store（`frontend/src/stores/auth.ts`）
- key：`auth.permissions`（Set\<string\>，O(1) 查询）
- 持久化：`localStorage`（刷新页面时恢复）

## 二、路由权限实现（router/index.ts）

### 路由守卫总览

```typescript
router.beforeEach((to, from, next) => {
    const auth = useAuthStore();
    const perm = usePermission();

    // 1. 登录态检查
    if (!auth.token && to.meta.requiresAuth !== false) {
        return next({ path: "/login", query: { redirect: to.fullPath } });
    }

    // 2. 路由 meta.permission 检查
    if (to.meta.permission && !perm.has(to.meta.permission as string)) {
        return next({ path: "/403" });
    }

    // 3. 动态菜单路由注册（如未注册过）
    if (!auth.dynamicRoutesRegistered) {
        registerDynamicRoutes(auth.menus);
        auth.dynamicRoutesRegistered = true;
        // 用 replace 触发一次重新匹配
        return next({ ...to, replace: true });
    }

    next();
});
```

### 路由定义示例

```typescript
// frontend/src/router/routes.ts
export const routes: RouteRecordRaw[] = [
    { path: "/login", component: () => import("@/views/Login.vue"), meta: { requiresAuth: false } },
    { path: "/403",   component: () => import("@/views/403.vue"),   meta: { requiresAuth: false } },
    {
        path: "/sys/user",
        component: () => import("@/views/sys/UserList.vue"),

            title: "用户管理",

        meta: {
            title: "文章管理",

            icon: "Document",

        }
    }
];
```

### 动态菜单路由注册

后端返回的菜单树（每个菜单项包含 `name/path/component/permission/children`）：

```typescript
// frontend/src/router/dynamic.ts
export function registerDynamicRoutes(menus: Menu[]) {
    for (const menu of flattenMenus(menus)) {
        if (!menu.component) continue;
        const route: RouteRecordRaw = {
            path: menu.path,
            name: menu.name,
            component: () => import(`@/views/${menu.component}.vue`),
            meta: {
                title: menu.title,
                permission: menu.permission,
                icon: menu.icon,
                breadcrumb: buildBreadcrumb(menu)
            }
        };
        router.addRoute("Layout", route);
    }
}
```

- 未在菜单中的路由不会被注册，从而无法访问（深度防御）。
- 每次刷新页面时，Pinia 重新加载菜单并调用 `registerDynamicRoutes`。

### 动态注册时机

| 时机 | 动作 |
|------|------|
| 登录成功 | 调用 `loadUserInfo()` → `registerDynamicRoutes(menus)` |
| 刷新页面 | Pinia 从 localStorage 恢复 → `registerDynamicRoutes(menus)` |
| 权限变更 | 管理员修改角色菜单 → 后端发送通知 → 前端重新加载菜单 |

### 路由 meta.permission 与后端对齐

前端路由中的 `meta.permission` 必须与后端 `sys_menu` 表中的 `permission` 字段对齐。

| 前端路由 meta.permission | 后端菜单 permission |
|--------------------------|---------------------|
| `"sys:user:view"` | `"sys:user:view"` |
| `"cms:article:edit"` | `"cms:article:edit"` |

## 四、按钮级权限（usePermission composable）

### usePermission composable 定义

位置：`frontend/src/composables/usePermission.ts`

```typescript
import { computed } from "vue";
import { useAuthStore } from "@/stores/auth";

export function usePermission() {
    const auth = useAuthStore();

    const permissions = computed(() => new Set(auth.permissions ?? []));

    function has(perm: string): boolean {
        if (!perm) return true;
        return permissions.value.has(perm);
    }

    function hasAny(perms: string[]): boolean {
        if (!perms?.length) return true;
        return perms.some(p => permissions.value.has(p));
    }

    function hasAll(perms: string[]): boolean {
        if (!perms?.length) return true;
        return perms.every(p => permissions.value.has(p));
    }

    function all(): string[] {
        return Array.from(permissions.value);
    }

    return { has, hasAny, hasAll, all };
}
```

### 在模板中使用：`v-permission` 指令

位置：`frontend/src/directives/permission.ts`

```typescript
import type { Directive, DirectiveBinding } from "vue";

function checkPermission(el: HTMLElement, binding: DirectiveBinding<string | string[]>) {
    const perm = usePermission();
    const arg = binding.arg;                         // "any" | "all"
    const value = binding.value as string | string[];

    let allow = false;
    if (Array.isArray(value)) {
        allow = arg === "any" ? perm.hasAny(value) : perm.hasAll(value);
    } else {
        allow = perm.has(value);
    }
    if (!allow && el.parentNode) {
        el.parentNode.removeChild(el);
    }
}

export const permission: Directive = {
    mounted: checkPermission,
    updated: checkPermission
};
```

注册指令（`main.ts`）：

```typescript
import { permission } from "@/directives/permission";
app.directive("permission", permission);
```

### 使用示例（模板内）

```vue
<template>
    <!-- 基础：有 sys:user:add 才显示此按钮 -->
    <a-button v-permission="'sys:user:add'" type="primary" @click="addUser">
        新增用户
    </a-button>

    <!-- 使用 v-if （更灵活，可与其他条件组合） -->
    <a-button v-if="perm.has('sys:user:edit')" @click="editUser">编辑</a-button>

    <!-- 组合权限：任意一个即可 -->
    <a-button v-permission:any="['cms:article:edit', 'cms:article:publish']">
        发布
    </a-button>

    <!-- 组合权限：必须全部拥有 -->
    <a-button v-permission:all="['cms:article:view', 'cms:article:publish']">
        高级发布
    </a-button>

    <!-- 按钮下拉菜单（使用 v-if） -->
    <a-dropdown>
        <template #overlay>
            <a-menu>
                <a-menu-item v-if="perm.has('sys:user:edit')">编辑</a-menu-item>
                <a-menu-item v-if="perm.has('sys:user:delete')" danger>删除</a-menu-item>
            </a-menu>
        </template>
        <a-button>操作 ▾</a-button>
    </a-dropdown>
</template>

<script setup lang="ts">
import { usePermission } from "@/composables/usePermission";
const perm = usePermission();
</script>
```

### 在脚本内使用（判断逻辑）

```typescript
const perm = usePermission();

async function handleAction(row: User) {
    if (perm.has("sys:user:delete")) {
        await confirm("确认删除此用户？");
        await deleteUser(row.userCode);
    } else {
        message.error("无权限删除用户");
    }
}
```

## 五、字段级权限

### 数据来源

`GET /api/v1/sys/auth/menu-route` 响应中，每个菜单包含 `fieldPermissions`：

```json
{
  "fieldPermissions": {
    "sys-user": [
      { "field": "phone",   "readable": true, "editable": false },
      { "field": "password","readable": false,"editable": false },
      { "field": "email",   "readable": true, "editable": true }
    ],
    "cms-article": [
      { "field": "author",  "readable": true, "editable": false },
      { "field": "viewCount","readable": true,"editable": false }
    ]
  }
}
```

### 列表页（隐藏列）

```vue
<template>
    <a-table :columns="displayedColumns" :data-source="users" />
</template>

<script setup lang="ts">
import { computed } from "vue";
import { useAuthStore } from "@/stores/auth";

const auth = useAuthStore();

const allColumns = [
    { key: "userCode", title: "账号" },
    { key: "userName", title: "姓名" },
    { key: "phone",    title: "手机" },
    { key: "password", title: "密码" } // 可能被隐藏
];

const displayedColumns = computed(() => {
    const fperm = auth.fieldPermissions["sys-user"] ?? [];
    const hiddenSet = new Set(fperm.filter(f => !f.readable).map(f => f.field));
    return allColumns.filter(col => !hiddenSet.has(col.key));
});
</script>
```

### 编辑页（禁用字段）

```vue
<template>
    <a-form :model="form">
        <a-form-item label="账号">
            <a-input v-model:value="form.userCode" disabled />
        </a-form-item>

        <a-form-item label="手机">
            <a-input v-model:value="form.phone" :disabled="!canEditPhone" />
        </a-form-item>

        <a-form-item label="邮箱">
            <a-input v-model:value="form.email" :disabled="!canEditEmail" />
        </a-form-item>
    </a-form>
</template>

<script setup lang="ts">
import { computed } from "vue";
import { useAuthStore } from "@/stores/auth";

const auth = useAuthStore();
const fperm = computed(() => {
    const list = auth.fieldPermissions["sys-user"] ?? [];
    return Object.fromEntries(list.map(f => [f.field, f]));
});
const canEditPhone = computed(() => fperm.value["phone"]?.editable ?? true);
const canEditEmail = computed(() => fperm.value["email"]?.editable ?? true);
</script>
```

### 后端双重保障

- 服务端返回用户对象时，`password`、`salt` 等敏感字段一律置为 `null` 或不返回。
- 提交修改时，若字段在权限表中为 `editable=false`，服务端会忽略该字段，即便前端被篡改也不会真正写入。

## 六、权限的刷新与同步

### 权限变更流程

```
管理员在"角色管理"修改角色权限
       │
       ▼
 后端更新 sys_role_menu / sys_role_permission
       │
       ▼
 Service 层调用 cacheService.RemoveAsync("auth:perm:*")
       │
       ▼
 Redis Pub/Sub 广播 Invalidate("auth:perm:*")
       │
       ▼
 所有应用节点清除 L1/L2 缓存 → 下次请求时从数据库重新加载
       │
       ▼
 受影响用户下次登录 / 刷新页面 → 重新 GET /api/v1/sys/auth/menu-route
```

### 前端刷新机制

| 场景 | 前端动作 |
|------|---------|
| 正常页面刷新 | Pinia 从 localStorage 恢复 → `registerDynamicRoutes()` |
| 主动刷新权限 | 右上角"刷新权限"按钮 → `await auth.loadUserInfo()` |
| 管理员通知推送（可选 WebSocket） | 收到推送后自动调用 `auth.loadUserInfo()` |

### loadUserInfo() 核心代码

```typescript
// frontend/src/stores/auth.ts
export const useAuthStore = defineStore("auth", {
    state: () => ({
        token: "",
        permissions: [] as string[],
        menus: [] as Menu[],
        fieldPermissions: {} as Record<string, FieldPermission[]>,
        dynamicRoutesRegistered: false
    }),
    actions: {
        async loadUserInfo() {
            const resp = await api.get<AuthInfo>("/sys/auth/info");
            this.permissions = resp.data.permissions;
            this.menus = resp.data.menus;
            this.fieldPermissions = resp.data.fieldPermissions;
            this.dynamicRoutesRegistered = false;
            registerDynamicRoutes(this.menus);
            this.dynamicRoutesRegistered = true;
        }
    },
    persist: { paths: ["token", "permissions", "menus", "fieldPermissions"] }
});
```

## 七、前端安全注意事项

### 1. 权限检查绝不能仅在前端完成

> ❌ 错误：仅用 `v-permission` 隐藏按钮，后端接口无权限校验
> ✅ 正确：`v-permission` 只是 UI 友好（不显示无权限按钮），**每个后端 API 必须二次校验**

### 2. 后端 API 权限校验示例

```csharp
// 后端 Filter / Attribute
[PermissionAuthorize("sys:user:delete")]
[HttpPost("{userCode}/delete")]
public async Task DeleteUser(string userCode)
{
    // ... 实际删除逻辑
}
```

### 3. Token 存储与安全

| 项 | 当前实现 | 说明 |
|----|---------|------|
| JWT 存储 | `localStorage` | 简单，受 XSS 风险 |
| 更安全方案 | `httpOnly` + `Secure` Cookie | 传统 Cookie 模式，可抵御 XSS 窃取 |

建议（生产环境可选方案）：

- 改为 `httpOnly Cookie` 模式承载 Token，由浏览器自动发送。
- 开启 CSP（Content Security Policy）限制脚本来源。
- 启用 `SameSite=Lax` 防止 CSRF。

### 4. 权限标识绝不混淆

| 规范 | 示例 |
|------|------|
| 格式 | `{模块}:{实体}:{动作}` |
| 模块 | `sys`、`cms`、`codegen`、`tasks`、`bpm`、`app` |
| 实体 | `user`、`role`、`menu`、`article`、`category` |
| 动作 | `view` / `add` / `edit` / `delete` / `export` / `import` / `publish` |

**不要在权限标识中使用中文或空格**。

### 5. 前端权限变更后的副作用

- 权限变更后必须重新注册动态路由。
- Pinia 的 `persist` 插件会自动写入 `localStorage`，刷新页面后自动恢复。
- 旧的路由缓存（`keep-alive`）应清理，避免用户看到旧页面。
