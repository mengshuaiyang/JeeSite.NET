本文档描述 JeeSite.NET 中最核心的两项业务安全能力——**数据权限（行级）**
与**字段权限（列级）**——如何与「角色-菜单」体系协同，形成"三层权限模型"。

---

### 一、权限体系三层结构

```
┌─────────────────────────────────────
│ 角色（sys_role）
│ 每个角色包含三层权限
│
├─ 1. 菜单权限（sys_role_menu → sys_menu）
│    控制：用户能看到哪些页面（前端路由 + 菜单树）
│
├─ 2. 数据权限（sys_role_data_scope + sys_menu_data_scope）
│    控制：用户能看到哪些数据行（SQL WHERE 注入）
│    类型：全部 / 本公司 / 本公司及以下 / 本机构 /
│          本机构及以下 / 仅本人 / 自定义 SQL
│
└─ 3. 字段权限（sys_role_field_scope）
     控制：用户能看到哪些字段列、能否修改
     类型：可见 / 只读 / 隐藏
```

- 三层权限均以 **角色** 为分配单位，用户通过"用户-角色"多对多获得叠加权限。
- **优先级规则**（取最高）：全部 > 本公司及以下 > 本公司 > 本机构及以下
  > 本机构 > 仅本人 > 自定义 SQL。
- 对拥有多个角色的用户，最终生效的数据权限是其所有角色中级别最高者。

### 二、数据权限类型详解

| 类型代码 | 名称 | 注入的 WHERE 条件 | 适用场景 |
|---------|------|-------------------|---------|
| `1` | 全部数据 | 无额外条件（查看全量数据） | 超级管理员、集团审计。 |
| `2` | 本公司数据 | `corp_code = @CurrentUser.corp_code` | 分公司管理员，仅管理自己公司。 |
| `3` | 本公司及以下 | `corp_code IN (SELECT corp_code FROM fn_child_corps(@CurrentUser.corp_code))` | 集团管理员，需要看到子公司全部数据。 |
| `4` | 本机构数据 | `org_code = @CurrentUser.org_code` | 部门经理，仅管理本部门员工与业务。 |
| `5` | 本机构及以下 | `org_code IN (SELECT org_code FROM fn_child_orgs(@CurrentUser.org_code))` | 多层组织管理，需要看到下属机构。 |
| `6` | 仅本人数据 | `create_by = @CurrentUser.user_code` | 普通员工、销售代表，仅看自己创建的数据。 |
| `7` | 自定义 SQL | `AND ( <自定义 SQL 片段> )` | 特殊业务场景，按规则过滤数据行。 |

#### 2.1 公司/机构递归函数

为支持"及以下"类型，系统提供两张内存表/CTE：

- `fn_child_corps(@corp_code)`：递归返回 `@corp_code` 及其所有子公司；
- `fn_child_orgs(@org_code)`：递归返回 `@org_code` 及其所有子机构。

两者均基于 `sys_company` 与 `sys_organization` 中的 `parent_code` 字段实现
标准邻接列表模型（最多 8 层，可配置）。

### 三、数据权限注入实现

#### 3.1 总体流程（EF Core QueryFilter + 全局拦截器）

```
HTTP 请求 → JWT 认证 → ClaimsPrincipal
   │
   ▼
 读取当前用户所有角色 → 计算最高数据权限类型（1..7）
   │
   ▼
 在仓储 / DbContext 层，针对"带数据权限"的实体动态注入 WHERE 条件：
   ├── 类型 1 → 不注入
   ├── 类型 2 → AND corp_code = @corpCode
   ├── 类型 3 → AND corp_code IN (SELECT ...)
   ├── 类型 4 → AND org_code = @orgCode
   ├── 类型 5 → AND org_code IN (SELECT ...)
   ├── 类型 6 → AND create_by = @userCode
   └── 类型 7 → AND ( <自定义 SQL> )
   │
   ▼
 结果返回前 → 字段权限过滤器（见第四节）
```

#### 3.2 关键实现点

1. **实体标记**：在数据实体上通过 `IDataScoped` 接口或 `[DataScope]` 特性
   标记该实体需要启用数据权限过滤；未标记的实体不受影响。
2. **EF Core QueryFilter**：`OnModelCreating` 中针对每个 `IDataScoped`
   实体注册 `HasQueryFilter`，使用 `CurrentUser.DataScopeType` 动态注入。
3. **全局拦截器**：手写 SQL / Dapper 场景下，`IDbCommandInterceptor`
   在执行前追加 WHERE 片段，确保不绕过。
4. **菜单级覆盖**：`sys_menu_data_scope` 允许对某个菜单指定与角色不同的
   数据权限（例如"销售管理员"角色默认"本公司"，但在"客户订单查询"菜单降级为
   "仅本人"），最终生效范围取"角色 vs 菜单"两者中更严格者。

#### 3.3 配置实体一览

| 表 | 作用 | 核心字段 |
|----|------|---------|
| `sys_role_data_scope` | 角色级数据权限配置 | `role_code`, `data_scope_type`, `custom_sql` |
| `sys_menu_data_scope` | 菜单级数据权限覆盖 | `menu_code`, `data_scope_type`, `sql_fragment` |
| `sys_user_data_scope` | （可选）用户级个性化覆盖 | `user_code`, `data_scope_type`, `custom_sql` |

**优先级**：用户级 > 菜单级 > 角色级（对同一用户同一菜单，取最严格者）。

### 四、字段权限实现

字段权限用于在同一行数据中，控制不同角色对各字段的可见性与可写性。

#### 4.1 配置表：sys_role_field_scope

| 列 | 说明 |
|----|------|
| `role_code` | 受配角色。 |
| `table_name` | 目标表名（对应实体名或视图名）。 |
| `field_name` | 目标字段名（对应列名或 DTO 属性名）。 |
| `permission_type` | `0`=可见（默认，不限制），`1`=只读，`2`=隐藏。 |
| `remark` | 备注（说明为何限制）。 |

唯一约束：`(role_code, table_name, field_name)`。

#### 4.2 服务端过滤机制

| 阶段 | 处理逻辑 |
|-----|---------|
| 查询返回（DTO 序列化前） | 对 `permission_type = 2`（隐藏）的字段，置为 `null` / `default`；对敏感字段还可写入审计日志记录访问者。 |
| 编辑/新增（写入前） | 校验 `permission_type`：若为 `1`（只读），拒绝更新该字段；若为 `2`（隐藏），拒绝写入（防止前端绕过）。 |
| 导出 Excel | 导出流中剔除 `permission_type = 2` 的列，避免外泄。 |

示例代码（Service 层过滤字段）：

```csharp
var list = await _userRepository.GetListAsync();
var hiddenFields = await _fieldScopeService
    .GetHiddenFieldsAsync(tableName: "sys_user", CurrentUser.Roles);

foreach (var item in list)
{
    foreach (var field in hiddenFields)
    {
        // 通过反射或 FastMember 将对应字段置为 default
        _fieldSetter.SetNull(item, field);
    }
}
```

#### 4.3 前端（Vue）集成

- `composables/useFieldScope.ts`：根据 `tableName` 拉取当前用户的字段权限
  （只读字段集合 / 隐藏字段集合）。
- `el-table` 渲染时：对 `hidden` 字段不渲染 `<el-table-column>`；
  对 `readonly` 字段渲染为纯文本列。
- `el-form` 表单中：对 `readonly` 字段使用 disabled 状态；对 `hidden` 字段
  不渲染表单项，同时服务端同步拦截，避免前端伪造提交。

#### 4.4 字段权限对 CodeGen 的自动支持

代码生成器（CodeGen）在生成前端表单与后端 DTO 时，会自动：

- 读取 `sys_role_field_scope` 中针对当前实体的限制；
- 在 DTO 上发出 `[FieldScope(roleCode, PermissionType)]` 元数据，
  由统一过滤器在运行时根据当前用户角色动态生效；
- 前端生成组件时自动按元数据处理 `v-if` / `:disabled`。

### 五、数据权限的 API 端点

#### 5.1 RoleDataScopeController

| 方法 | 路径 | 说明 |
|-----|------|------|
| `GET` | `/api/v1/sys/role-data-scope/{roleCode}` | 查询指定角色的数据权限配置。 |
| `PUT` | `/api/v1/sys/role-data-scope/{roleCode}` | 更新指定角色的数据权限配置。 |

| 方法 | 路径 | 说明 |
|-----|------|------|

| `PUT` | `/api/v1/sys/role-field-scope/{roleCode}/{tableName}` | 更新角色对某表的字段权限。 |

#### 5.3 MenuDataScopeController

| 方法 | 路径 | 说明 |
|-----|------|------|

| `PUT` | `/api/v1/sys/menu-data-scope/{menuCode}` | 更新某菜单的覆盖配置。 |

### 六、最佳实践

1. **设计时就考虑数据隔离粒度**：核心业务实体务必带上 `corp_code / org_code /
   create_by` 三字段，使得数据权限机制无需额外改造即可接入。

2. **字段权限不要过度精细**：控制粒度建议到"是否能看到工资、电话、身份证"等
   敏感字段级别即可，过多限制会增加开发与调试成本，并可能造成接口性能退化。

3. **自定义 SQL 片段必须经过安全审查**：`custom_sql` / `sql_fragment` 为
   纯 SQL 片段，存在注入风险，应当：
   - 限制写入权限（仅管理员）；
   - 入库前做白名单校验（仅允许 `AND field IN (...)`、`AND field = @CurrentUser.xxx`
     等可控模式）；
   - 禁止引入字符串拼接，所有动态值通过参数化传入。

4. **数据权限是"AND"叠加**，多角色时取最高权限：用户若拥有多个角色，系统使用
   最高数据权限（最宽松），而不是 OR 合并，避免普通角色+管理角色产生越权漏洞。

5. **对 CodeGen 生成代码自动生效**：所有新业务模块，只要实体实现
   `IDataScoped`，自动享受数据/字段权限能力，无需模块级额外开发。

6. **审计日志不可少**：对字段权限命中"隐藏"的访问、对数据权限被"自定义 SQL"
   过滤的请求，应独立记录审计日志，便于后续审计。

7. **性能注意**：递归子公司/子机构查询应尽量走内存缓存（`corp_code_tree`），
   避免每次请求都递归 DB；对"及以下"类型，可在 Redis 中预计算并设置 10 分钟
   TTL，组织架构变更时主动失效。

### 七、示例：一个销售角色的完整配置

以「区域销售经理」角色为例，三层权限的典型配置如下：

```
角色：区域销售经理（role_code = "area_sales_manager"）

  1. 菜单权限：销售管理全部菜单
     sys_role_menu.role_code = "area_sales_manager"
     sys_role_menu.menu_code IN ("sales.overview", "sales.order", ... )

  2. 数据权限：本机构及以下（类型 5）
     sys_role_data_scope.role_code = "area_sales_manager"
     sys_role_data_scope.data_scope_type = 5

  3. 字段权限：
     sys_role_field_scope.role_code = "area_sales_manager"
       ├── table: sales_order
       │     ├── 字段 customer_mobile  → 只读 (1)
       │     └── 字段 internal_cost   → 隐藏 (2)
       └── table: customer_profile
             └── 字段 id_card_no     → 隐藏 (2)
```

最终效果：区域销售经理在销售管理模块，能看到本机构及下属机构所有销售订单数据；
但在订单列表中无法看到「内部成本」列，也无法修改客户手机号；即使通过浏览器
开发者工具修改页面 DOM 提交，服务端也会二次校验拦截。

---

**第五部分小结**：数据权限与字段权限是 JeeSite.NET 在"角色-菜单"之上的两层核心能力。
通过在 **DbContext QueryFilter / Service 层 / Vue 组件** 三个层面的统一机制，
业务模块无需额外开发即可实现行级与列级访问控制，配合审计日志可满足严格的合规要求。
